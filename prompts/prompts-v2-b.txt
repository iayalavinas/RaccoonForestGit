# Generate dependency goals
Next, in the dependency, describe only the goals that each actor wants for the other actors and
add them to the JSON as described above. Do not describe goals in nodes within each actor.
Note that the source and target of the dependency is the one requesting the dependency element
as the source and the one providing the dependency element as the target.
Note: Ensure that the ”source” and ”target” belong to different actors. Dependencies within
the same actor are not allowed.
Description Rules for Dependency Links:
Rule1:The type of dependencies can only be one of the following: iStar.Goal, iStar.Task, iStar.Resource, or iStar.Quality.

# Generate goals within actors
Next, within each actor, describe the goals necessary to achieve the dependent goals, as well as
any domain-specific goals derived from the client’s detailed requirements.

# Generate tasks within actors
Next, Identify and create the iStar.Tasks required to achieve the iStar.Goals within each actor’s
node, and connect the created tasks to the parent iStar.Goal using either AND Refinement Links
or OR Refinement Links. The connections made with AND Refinement Links and OR Refinement Links between iStar.Goals and iStar.Tasks must strictly adhere to the Link Description
Rules.
When creating an iStar.Task, follow these steps as a reference.
Step 1: Create the iStar.Task.
Step 2: Link the iStar.Task to an iStar.Goal.
Link Description Rules for iStar.Task：
Rule 1:AND Refinement starts from Task with the arrowhead pointing to Goal.
Rule 2:OR Refinement starts from Task with the arrowhead pointing to Goal.
Rule 3:AND Refinement starts from Task with the arrowhead pointing to Task.
Rule 4:OR Refinement starts from Task with the arrowhead pointing to Task.
Rule 5:The source and target of an AND-refinement link must pertain to the same actor.
Rule 6:You cannot mix AND-refinements with OR-refinements targeting the same element.
Rule 7:The source and target of an OR-refinement link must pertain to the same actor.

# Generate resources within actors
Next, Create the iStar.Resources required for the iStar.Tasks within each actor’s node, and
connect the created iStar.Resources to their respective iStar.Tasks using Needed-by Links. There
is no need to consider iStar.Resource for an iStar.Goal. When creating an iStar.Resource, follow
these steps as a reference. The connections made with Needed-by Links between iStar.Tasks and
iStar.Resources must adhere to the Link Description Rules for iStar.Resource.
Step 1: Create the iStar.Resource.
Step 2: Link the iStar.Resource to an iStar.Task.
Link Description Rules for iStar.Resource：
Rule 1:The source of a NeededBy can only be set to an iStar.Resource, and the target can only
be set to an iStar.Task.
Rule 2:The source and target of a Needed-By link must pertain to the same actor.
Rule 3:iStar.Goal and iStar.Resource must not be connected using a Needed-by link. If you want
to connect an iStar.Resource to an iStar.Goal using a Needed-by link, first create an iStar.Task
as a child element of the iStar.Goal you want to connect. Then, connect the newly created
iStar.Task to the iStar.Goal using an AndRefinementLink or OrRefinementLink. After that,
connect the newly created iStar.Task and the iStar.Resource using a Needed-by link.
Rule 4:Create the iStar.Resource only for iStar.Tasks and not directly for iStar.Goals.
Rule 5:Always create an iStar.Task before connecting a Resource to a Goal.

# Generate qualities within actors
Next, Identify the necessary non-functional requirements from the client’s detailed requirements
and create them as iStar.Quality. Place the created iStar.Quality within the node of the actor containing the related elements, and connect it to the relevant elements using either an
iStar.Qualification Link or an iStar.Contribution Link. When using an iStar.Contribution Link,
ensure that the ”label” is specified and contains one of the following: Make, Help, Hurt, or
Break. Note that for links related to iStar.Quality, the iStar.Quality element must always be the
target. Connections between iStar.Quality and the relevant elements using iStar.Qualification
Links or iStar.Contribution Links must strictly adhere to the following Link Description Rules.
Link Description Rules for iStar.Quality：
Rule 1:Qualification starts from Quality with the arrowhead pointing to Quality in the actor.
Rule 2:Contribution starts from Quality with the arrowhead pointing to Goal in the actor.
Rule 3:Qualification starts from Quality with the arrowhead pointing to Task in the actor.
Rule 4:Qualification starts from Quality with the arrowhead pointing to Resource in the actor.
Rule 5:Contribution starts from Task with the arrowhead pointing to Quality in the actor.
Rule 6:Contribution starts from Resource with the arrowhead pointing to Quality in the actor.
Rule 7:Contribution Links must have four types (Make, Help, Hurt, Break) and are described
in the “label” tag in the JSON text. If an element is considered to contribute sufficiently to a
certain quality, “Make” is used. If an element is considered to contribute weakly to a certain
quality, “Help” is described. Hurt is used when an element weakly negates a certain quality.
Break is used when an element sufficiently negates a certain quality. (For example, ”type”:
”istar.ContributionLink”, ”source”: ”Agency buys tickets(3.8)”, ”target”: ”Minimal own payments(3.13)”, ”label”: ”help” )
Rule 8:The source and target of a Contribution link must pertain to the same actor.
Rule 9:The source and target of a Qualification link must pertain to the same actor.

# Generate root goals within actors
These instructions apply to the elements (iStar.Goal, iStar.Task) within the actor’s node, and no
transformation should be applied to the actor itself (iStar.Actor), such as its ”text”. Within the
node of an actor, if there are multiple root elements (iStar.Goal, iStar.Task), create a new goal
that acts as the parent element for all these root elements. Link all existing root elements to this
new parent goal appropriately. When linking the new parent element with the root elements,
adhere to the following Link Description Rules.
Link Description Rules:
Rule 1:AND Refinement starts from Task with the arrowhead pointing to Goal.
Rule 2:OR Refinement starts from Task with the arrowhead pointing to Goal.
Rule 3:AND Refinement starts from Task with the arrowhead pointing to Task.
Rule 4:OR Refinement starts from Task with the arrowhead pointing to Task.
Rule 5:The source and target of an AND-refinement link must pertain to the same actor.
Rule 6:You cannot mix AND-refinements with OR-refinements targeting the same element.
Rule 7:The source and target of an OR-refinement link must pertain to the same actor.

# Modify the dependency’s sources and targets to reside within actors
Next, change the source and target of the dependency to nodes within the actor, clarifying the
dependency. Modify the source of the dependency to an appropriate leaf element within the
actor currently set as the source. A leaf element refers to an element that has never been set as
the target of an AndRefinementLink or OrRefinementLink. Adjust the target of the dependency
to an appropriate node element within the actor currently set as the target. The configuration

Description Rules for Dependency Links:
Rule1:Elements set as targets in AND Refinement and OR Refinement cannot be sources in
Dependency Links.
Rule2:The source and target of a Dependency link must not pertain to the same actor.
An incorrect example of Rule 2:
"dependencies": [{ "text": "Dependency", "type": "istar.Goal",
"source": "Node B of Actor A(n1.3)", "target": "Node C of Actor A(n1.1)" },]
Rule3:The type of dependencies can only be one of the following: iStar.Goal, iStar.Task, iStar.Resource, or iStar.Quality.
Ensure that the source actor id and target actor id in each dependency are different.
A dependency between the same actor is invalid.
